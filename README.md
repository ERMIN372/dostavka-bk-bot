# dostavka-bk-bot

Внутренний Telegram-бот для сотрудников службы доставки БК компании
**«ТД Нефтьмагистраль»**. Отвечает на вопросы по инструкции доставки на основе
PDF-базы знаний с помощью RAG (retrieval-augmented generation): находит
релевантные фрагменты инструкции локальным семантическим поиском и генерирует
ответ через **YandexGPT**.

## Как это работает

1. При старте бот индексирует все PDF из папки `knowledge_base/`: извлекает
   текст (PyMuPDF), распознаёт текст на изображениях/схемах страниц (OCR,
   tesseract-rus) и приклеивает его к тексту той же страницы, режет всё на чанки
   с перекрытием и считает эмбеддинги локальной моделью
   `intfloat/multilingual-e5-large`.
2. Индекс кешируется в `knowledge_base/.cache/`. Если PDF не менялись (совпадает
   хеш) — при следующем старте индекс просто загружается из кеша, без пересчёта.
3. На каждый вопрос: считается эмбеддинг вопроса → ищутся top-5 чанков по
   косинусному сходству. Если лучший чанк ниже порога сходства — бот сразу
   отвечает «не найдено» и **не** дёргает YandexGPT (экономия). Иначе найденные
   чанки идут в промпт, и YandexGPT формирует ответ строго по ним.

Особенности по требованиям проекта:
- Никакой БД (ни SQLite, ни Postgres), никаких векторных БД. Поиск — numpy в памяти.
- Вопросы пользователей **не логируются**. В stdout идут только служебные логи.
- Памяти диалога нет: каждое сообщение обрабатывается независимо.
- Добавление новых PDF не требует изменения кода — просто кладёте файл в папку.

## Структура репозитория

```
knowledge_base/            PDF-файлы базы знаний (сюда кладётся исходный PDF)
knowledge_base/.cache/     предпосчитанный индекс (эмбеддинги + тексты + хеш файлов)
src/pdf_processor.py       извлечение текста + OCR изображений, чанкинг с overlap
src/indexer.py             построение/загрузка/кеширование индекса, проверка по хешу
src/retriever.py           top-k поиск по косинусному сходству + порог схожести
src/yandex_gpt.py          обёртка над YandexGPT API
src/bot.py                 aiogram-бот, обработчики сообщений
main.py                    точка входа: строит/загружает индекс и запускает polling
requirements.txt
Procfile / railway.json / nixpacks.toml   конфиги для Railway (worker-процесс)
.env.example
```

## Переменные окружения

| Переменная | Обязательна | Описание |
|---|---|---|
| `TELEGRAM_BOT_TOKEN` | да | Токен бота от [@BotFather](https://t.me/BotFather) |
| `YANDEX_API_KEY` | да | API-ключ (`Api-Key`) сервисного аккаунта Yandex Cloud |
| `YANDEX_FOLDER_ID` | да | Идентификатор каталога (folder) в Yandex Cloud |
| `YANDEX_GPT_MODEL` | нет | Модель YandexGPT, по умолчанию `yandexgpt-lite` |
| `EMBEDDING_MODEL_NAME` | нет | Модель эмбеддингов, по умолчанию `intfloat/multilingual-e5-large` |
| `KNOWLEDGE_BASE_DIR` | нет | Путь к папке базы знаний, по умолчанию `./knowledge_base` |

Если `YANDEX_API_KEY`, `YANDEX_FOLDER_ID` или `TELEGRAM_BOT_TOKEN` не заданы —
бот **явно падает при старте** с понятным сообщением в логах (а не в рантайме на
первом сообщении пользователя).

### Где взять ключи

- **`TELEGRAM_BOT_TOKEN`** — напишите [@BotFather](https://t.me/BotFather),
  команда `/newbot`, следуйте инструкции, скопируйте выданный токен.
- **`YANDEX_API_KEY`** и **`YANDEX_FOLDER_ID`** — в консоли
  [Yandex Cloud](https://console.yandex.cloud/): создайте сервисный аккаунт,
  выдайте ему роль `ai.languageModels.user`, создайте для него **API-ключ**
  (это `YANDEX_API_KEY`). `YANDEX_FOLDER_ID` — идентификатор каталога, виден на
  странице каталога в консоли.

> На момент старта проекта сервисного аккаунта ещё нет — код полностью готов к
> работе, ключи вставляются в переменные окружения, когда аккаунт появится.

## Локальный запуск

Нужен Python 3.11+ и системный **tesseract** с русским языковым пакетом.

```bash
# 1. Системная зависимость OCR
#   Ubuntu/Debian:
sudo apt-get install tesseract-ocr tesseract-ocr-rus
#   macOS (Homebrew):  brew install tesseract tesseract-lang

# 2. Виртуальное окружение и зависимости
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 3. Ключи
cp .env.example .env      # и впишите свои значения

# 4. Положите PDF с инструкцией в knowledge_base/

# 5. Запуск
python main.py
```

При первом запуске модель эмбеддингов (~1–2 ГБ) скачается автоматически, а индекс
посчитается с нуля — это может занять от десятков секунд до пары минут. Дальше,
если PDF не менялись, индекс берётся из кеша.

## Добавление документов в базу знаний

Поддерживаются два формата — просто положите файл в `knowledge_base/` и
перезапустите бота. При старте бот заметит, что набор файлов изменился (по
хешу), переиндексирует базу и обновит кеш. **Менять код не нужно.**

- **`.md` (рекомендуется)** — Markdown в формате Q&A: `## Раздел`, `### Вопрос`,
  под ним текст ответа. Каждая пара «вопрос-ответ» становится отдельным чанком —
  это даёт максимальную точность поиска. Строки `*Источник: …*` игнорируются.
  Эталон: `knowledge_base/BK_delivery_knowledge_base_QA.md`.
- **`.pdf`** — для документов, которых нет в текстовом виде: извлекается
  текстовый слой + OCR изображений, чанкинг по страницам.

Не кладите один и тот же материал в двух форматах одновременно — дубли тем
конкурируют в поиске и ухудшают качество ответов.

## Деплой на Railway

Бот настроен как **worker-процесс** (не web-сервис): публичный HTTP-эндпоинт не
нужен, используется long polling.

1. Создайте проект в Railway из этого репозитория.
2. В **Variables** задайте `TELEGRAM_BOT_TOKEN`, `YANDEX_API_KEY`,
   `YANDEX_FOLDER_ID` (и при желании `YANDEX_GPT_MODEL`).
3. Сборка идёт через Nixpacks (`nixpacks.toml`), который ставит системный
   `tesseract-ocr` + `tesseract-ocr-rus`. Запуск — `python main.py`
   (`Procfile` / `railway.json`).

> **Важно про кеш и передеплой.** При каждом передеплое Railway поднимает
> контейнер с нуля. Папка `knowledge_base/.cache/` живёт на эфемерном диске
> контейнера, поэтому без персистентного **volume** индекс будет
> пересчитываться при каждом деплое. Для одного 84-страничного PDF это
> ожидаемо и занимает секунды/десятки секунд на CPU — сюрприза быть не должно.
> Если захотите избежать пересчёта — подключите volume и примонтируйте его на
> путь `knowledge_base/.cache` (или на всю `knowledge_base`).

## Калибровка порога similarity threshold

Порог задан константой `SIMILARITY_THRESHOLD` в `src/retriever.py` (стартовое
значение `0.5`). Он определяет, при каком минимальном сходстве бот вообще идёт в
YandexGPT, а не отвечает «Не нашёл ответа…».

Как понять, что порог настроен неверно, и что делать:

- **Бот слишком часто отвечает «Не нашёл ответа», хотя ответ в инструкции есть** —
  порог слишком высокий. **Понизьте** его (например, до `0.45` / `0.4`).
- **Бот отвечает мимо темы / выдаёт ответ на нерелевантные вопросы** — порог
  слишком низкий. **Повысьте** его (например, до `0.6` / `0.7`).

Как калибровать на практике: в логах при каждом запросе печатается
`Retrieval: max_score=... threshold=...`. Задайте боту 15–20 реальных вопросов
сотрудников (и несколько заведомо «не по теме»), посмотрите значения `max_score`
для тех, где ответ *должен* находиться, и для тех, где *не должен*. Выберите
порог между этими двумя группами. Для модели e5 «релевантные» сходства часто
лежат в районе `0.75–0.85`, так что итоговый порог, скорее всего, будет выше
стартового `0.5` — обязательно проверьте на своих данных.
